# App Hub Security Checklist

Date checked: 2 May 2026

## Source and package hygiene

- [x] No committed `.env` files were found in `rade-dhis2-app`.
- [x] No credentials, API keys, tokens, or passwords are required by the frontend app.
- [x] The demo fixtures are synthetic and contain no real patient data or PHI.
- [x] The build artifact is generated by `d2-app-scripts build` and runs inside the DHIS2 App Platform shell.
- [x] The app is intended for public source review at `https://github.com/EmilAP/rade-for-dhis2`.

## Network and external services

- [x] No external AI service is required.
- [x] The local decision engine is deterministic and runs in the browser.
- [x] DHIS2 Tracker validation uses the current DHIS2 App Runtime session through `@dhis2/app-runtime`.
- [x] No direct browser calls to credentialed third-party services are used by the default demo workflow.
- [ ] Optional future route-based RaDE integration must be reviewed before enabling in production; it should use DHIS2 Routes or another server-side proxy so secrets are not exposed to the browser.

## Frontend safety checks

- [x] No `dangerouslySetInnerHTML` usage was found.
- [x] External links opened with `target="_blank"` use `rel="noreferrer"`.
- [x] No external scripts, stylesheets, or CDNs are added by the app source.
- [x] Form controls have accessible labels or aria labels where straightforward.
- [x] Long raw JSON/debug output is collapsed by default.

## DHIS2 platform behavior

- [x] Uses DHIS2 App Platform tooling (`d2-app-scripts`).
- [x] Uses `@dhis2/app-runtime` for DHIS2 API access.
- [x] Uses `@dhis2/ui` components for app shell controls and cards.
- [x] The app does not ask users to paste DHIS2 usernames or passwords.
- [x] DataStore namespace is documented as `rade-dhis2`; key is `config`.
- [x] DataStore config is optional for demo mode and contains non-secret runtime defaults only.

## Metadata and production guardrails

- [x] Demo mode can run without importing DHIS2 metadata.
- [x] Default `RADE_*` and `PLACEHOLDER_*` metadata IDs are treated as scaffold values.
- [x] Mapping/readiness checks flag placeholder IDs before validation/import.
- [x] Production use requires local Tracker metadata mapping, DHIS2 validation-mode testing, role-based access review, and governance sign-off.
